1. Идентифтикация проблем

Ниже описаны проблемы и кратко пути решения:

1.1. Бизнес-проблемы.

Проблема Б1.
 Из " Клиенты жалуются менеджерам по продажам, что они не получили заказ и над их изделием работают уже несколько месяцев, 
хотя обещали закончить за три недели. 
Проблемы не на стороне производства: производственные мощности позволяют справиться с заказами и 
могут обеспечить двукратный рост."
следует, что менеджеры по продажам действуют реактивно (по жалобам). --> Дать менеджерам по продажам аналитику по текущим долгоиграющим/крупным заказам - какие повисли в каких статусах

Проблема Б2.
Из "- компания открыла свой API. жалоб от клиентов на просроченные заказы стало ещё больше.
 Добавились жалобы со стороны пользователей API"
следует, что команда не управляет сроком жизни заказов. 

 Проблема Б3.
 " - жалобы от операторов: когда они заходят на первую страницу MES, система долго прогружается."
 --> должна быть предподготовленная информация

1.2. Технические проблемы

Проблема Т1. Базы данных имеют один инстанс, который работает на запись и на чтение. --> для нагрузки чтения сделать отдельный инстанс
Проблема Т2. "В release-окружения версию деплоят вручную." --> нет CD
Проблема Т3. "Каждый из инстансов крутится как EC2, базы данных — это Managed services на Yandex Cloud. " --> БД далеко от логики

Проблема Т4. Нет единого сервиса по ведению Заказов. Информиция по зачазам лежит как в БД Shop, так и БД MES. 
--> Создать микросервис заказы. В него вынести логику обработки и правила ЖЦ заказов.

Проблема Т5. Абсолютно отсутствует - Мониторинг. Логирование. Трассировки.
--> повесить системный мониторинг -> Kibana
--> повесить RED логирование --> развернуть ELK
--> Развернуть OpenTelemetry и Jaegger

Проблема Т6. Внешние пользователи - это всегда непредсказуемая нагрузка. --> Должен стоять ограничитель трафика.

Проблема Т7. API для внешних пользователей возможно синхронное, что приводит к длительным ответам и эффекту зависания. 
  --> сделать асинхроннымм. 

1.3. Проблемы процесса

Проблема П1. Окружений всего три: dev, release и prod. 
  --> добавить окружение для автотестов
  --> добавить окружение для ручного тестирования (release использовать только для тестирования разворачивания и финальных тестов "как на проде")

Проблема П2. "Деплой в dev-окружение осуществляется автоматически после мёрджа пул-реквеста и прохождения юнит-тестов." 
 --> нет интеграционных тестов

Проблема П3. "Команда выпускает софт двухнедельными спринтами. ...  Раньше задержки не превышали одного месяца, но в последнее время требуется всё больше и больше времени."
 --> разбить систему на микросервисы
 --> автотесты 
 --> release в процессе тестов будет сильно отличаться от ПРОМа
 --> команды надо разбить по сервисам 

Проблема П4. Команда подключает новых массовых потребителей  (API), но нет рассчета нагрузки и оценки capacity.
-->  Провести нагрузочное тестирование на поиск максимума системы, оценки надежности. 

2. ИНИЦИАТИВЫ

  Здесь сперва идет индекс проблем(ы), затем инициатива. Указаны в порядке приоритета - Тушение->Стабилизация->Развитие.
Внутри раздела инициативы также отсортированы по приоритетам. 

2.1. Тушение Пожара

 - Б3. Закэшировать данные для первой страницы MES.
 - Т3. БД и логику перенести в единое облако.
 - Т5. Мониторинг RED. Настройка Prometheus и Grafana
 - Т5. Системный мониторинг (Prometheus и Grafana)
 - Т5. Логирование. Настройка ELK
 - T6. Настроить Rate Limiter для вызовов API

2.2. Стабилизация

- Т2. Настроить автодеплой во все окружения.
- Т4. Создать микросервис Заказы. В него вынести логику обработки и правила ЖЦ Заказов.
- Т1. Выделить реплику в PostreSQL на чтение
- Б1/Б2. Создать аналитическую витрину (возможно read-only ноду БД). Это даст менеджерам по продажам аналитику по текущим долгоиграющим/крупным заказам - какие повисли в каких статусах
- Т5. Трейсинг. Настройка Jaegger
- Т7. Перевести MES API на асинхронную обработку через Kafka.

2.3. Развитие

- П1/П2. Добавить окружение для интеграционных автотестов
- П3. Выделить команды и разбить по сервисам 
- П3. Разбить систему на микросервисы
- П1. Выделить окружение для ручного тестирования, release использовать только для тестирования разворачивания и финальных тестов "как на проде")  
- П4. Сформировать практику нагрузочного тестирования на поиск максимума системы, оценки надежности.

3. ВОПРОСЫ

 - Какой вы видите целевую архитектуру через полгода?

  Архитектура с одной стороны позволяет проактивно реагировать Продажам на зависшие сервисы, 
 но с другой стороны, сама архитектура уменьшает количество зависаний. 
 Обязательно выполнены все пункты из разделов "Пожар" и "Стабилизация" 

 - Если бы у вас была возможность выполнить только три пункта из списка инициатив в ближайшие полгода, 
 что бы вы выбрали и почему?

  1. Настроил бы мониторинг и логирование - чтобы иметь возможность идентифицировать проблемы. 
 И иметь возможность обосновывать и планировать дальнейшие изменения.
  2. Создать микросервис Заказы. В него вынести логику обработки и правила ЖЦ Заказов. -- это позволит прекратить текущий хаос с Заказами.
  3. Перевести MES API на асинхронную обработку через Kafka. -- это позволит не завалить систему нагрузкой от открытого АПИ.

   Эти решения не позволят увеличить прибыль, но позволят протянуть полгода на текущих объемах обслуживания.
