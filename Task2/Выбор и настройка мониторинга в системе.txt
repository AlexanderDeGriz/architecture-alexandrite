1. Мотивация

Внедрение мониторинга позволит решить следующие задачи:

1.1. Выявление узких мест в производительности:

Анализ нагрузки на сервисы (например, CRM, MES, онлайн-магазин) покажет, какие компоненты требуют оптимизации или масштабирования.

1.2. Отслеживание проблем с заказами.

Клиенты жалуются на просроченные заказы, а представители компаний-партнёров сообщают о потерянных заказах. Мониторинг позволит отслеживать статусы заказов в реальном времени, выявлять "зависшие" или необработанные заказы, а также анализировать задержки на каждом этапе (например, расчёт стоимости, подтверждение в CRM, производство).

1.3. Контроль работы API

После открытия API для внешних продавцов нагрузка на систему резко возросла, что привело к увеличению жалоб. Мониторинг API-запросов поможет:
 - Определять частоту и время выполнения запросов
 - Выявлять ошибки (например, таймауты, неудачные расчёты стоимости)
 - Обеспечивать SLA для партнёров

1.4. Оптимизация взаимодействия между системами:

 Интеграция между CRM и MES через RabbitMQ может быть источником проблем (например, потеря сообщений, задержки). Мониторинг очередей сообщений покажет:
  - Количество сообщений в очередях
  - Время обработки сообщений
  - Ошибки при передаче данных

1.5. Предотвращение потери клиентов:

Компания уже потеряла несколько крупных контрактов из-за проблем с заказами. Мониторинг поможет оперативно реагировать на критические ситуации (например, сбои в MES или CRM) до того, как они повлияют на клиентов. Что даст компании мониторинг:
 - Возможность проактивного анализа 
 - Снижение количества жалоб за счет раннего обнаружения и устранения проблем 
 - Прозрачность процессов: команда сможет видеть реальное состояние системы, что упростит поиск причин проблем и их устранение.
 - Оптимизация ресурсов: данные мониторинга помогут обосновать необходимость масштабирования
 - Анализ метрик (например, среднее время выполнения заказа, нагрузка на API) поможет бизнесу планировать рост и оценивать риски.


2. Выбор подхода к мониторингу

2.1. Какой подход к мониторингу использовать: 

1. CRM, Онлайн-магазин, Shop API, MES API -> RED 
2. MES -> USE (фокус на ресурсах и производительности)
3. RabbitMQ -> USE + RED (контроль очередей и интеграции)
4. БД -> USE (фокус на ресурсах и производительности) 

2.2. Какие метрики отслеживать:

Number of dead-letter-exchange letters in RabbitMQ - отслеживать количество неудачных попыток. Цель - убрать проблемы с необработанными заявками и расхождениями
Number of message in flight in RabbitMQ - количество сообщений, которые были отправлены потребителю (consumer), но ещё не подтверждены и не отклонены. - отслеживать количество неудачных попыток. Цель - анализ насыщения

Для следующих метрик цель - анализ общей входящей нагрузки
Number of requests (RPS) for internet shop API
Number of requests (RPS) for CRM API
Number of requests (RPS) for MES API

Для следующих метрик цель - анализ перекосов по входящей нагрузки от одного пользователя
Number of requests (RPS) per user for internet shop API
Number of requests (RPS) per user for CRM API
Number of requests (RPS) per user for MES API

Для следующих метрик цель - анализ системных показателей ресурсов платформы
CPU % for shop API
CPU % for CRM API
CPU % for MES API
Memory Utilisation for shop API
Memory Utilisation for CRM API
Memory Utilisation for MES API
Memory Utilisation for shop db instance
Memory Utilisation for MES db instance

Можно отслеживать, так как превышение количества коннектов к PG может вылиться в ошибки (но используем PG bouncer)
Number of connections for shop db instance
Number of connections for MES db instance

Надо отслеживать, чтобы сетевые задержки не оказывали влияние на скорость АПИ
Response time (latency) for shop API
Response time (latency) for CRM API
Response time (latency) for MES API

Наблюдаем за свободным местом под 3d файлы
Size of S3 storage

Наблюдаем за рамером БД. В целом необязательно, поскольку БД у Yandex, но может привести к ошибкам
Size of shop db instance
Size of MES db instance

Наблюдаем, чтобы оценить процент и абсолютное значение неуспешных 
Number of HTTP 200 for shop API
Number of HTTP 200 for CRM API
Number of HTTP 200 for MES API
Number of HTTP 500 for shop API
Number of HTTP 500 for CRM API
Number of HTTP 500 for MES API
Number of HTTP 500 for shop API

Наблюдаем, чтобы не пропустить проблем с подключениями
Number of simultanious sessions for shop API
Number of simultanious sessions for CRM API
Number of simultanious sessions for MES API

Следим за размером деталей
Kb tranferred (received) for MES API
Kb provided (sent) for MES API

3. План действий

3.1. Развёртывание инфраструктуры мониторинга

 - Развернуть Prometheus для сбора метрик с MES, CRM, RabbitMQ и онлайн-магазина.
 - Настроить Grafana для визуализации метрик и создания дашбордов.
 - Добавить алерты в Grafana для настройки алертов на критические события (например, высокая загрузка CPU, нехватка памяти, ошибки в очередях).

3.2. Доработка сервисов инструментальными средствами:

 - Внедрить экспортеры метрик для RabbitMQ, pg_exporter для PostgreSQL, Java-приложений (Micrometer/Prometheus JMX exporter),.NET-приложений (Prometheus.NET).

3.3. Настроить сбор логов через ELK-стек (Elasticsearch + Logstash + Kibana).

3.4. Добавить трейсинг (OpenTelemetry + Jaeger) для отслеживания транзакций между MES, CRM и RabbitMQ.
