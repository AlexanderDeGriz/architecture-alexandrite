1. Мотивация

Логирование — это систематическая запись событий и действий в ИТ-системах. Оно необходимо для:
 - Отслеживания работы системы в реальном времени.
 - Быстрого выявления и диагностики проблем (например, потерянных заказов или сбоев в MES).
 - Анализа производительности и поиска узких мест.
 - Соответствия регуляторным требованиям (аудит, ФЗ-152).
Без логирования компания действует вслепую: жалобы клиентов растут, а причины проблем остаются неизвестными.

1.1. Пример Технических метрик

1.1.1. Скорость обработки заказов
 
   Логи покажут, на каких этапах (MES, RabbitMQ) возникают задержки.	Уменьшение времени выполнения заказов.

1.1.2. Стабильность интеграций

  	Логи RabbitMQ выявят потерянные сообщения между CRM и MES.	Снижение количества "потерянных" заказов.

1.1.3. Частота ошибок API	

  Логи API помогут отследить, какие запросы от внешних продавцов приводят к 5xx-ошибкам.	Повышение надежности API и сохранение клиентов.

1.2. Бизнес-метрики

1.2.1. Удовлетворенность клиентов	

  Логи выявят частые причины жалоб (например, задержки в MES).	Снижение негативных отзывов и возвратов.

1.2.2. Конверсия заказов	

  Анализ логов покажет, на каком этапе (например, оплата или расчёт стоимости) клиенты отваливаются.	Увеличение завершенных покупок.

1.3. Для каких систем нужно настраивать логирование и трейсинг в первую очередь и почему.

  Прежде всего логирование и трейсинг надо настраивать на Rabbit MQ, на MES и на MES API - поскольку это "рабочие" модули, приносящие ценность Компании

2. Предлагаемое решение

Для внедрения логов надо внедрить: ELK Stack ( Elasticsearch + Logstash + Kibana) . Filebeat использовать в качестве агента.
См. предлагаемую схему в jewerly_c4_model___added_logging.drawio

3. Безопасность

Для безопасности хранения трейсинга внутри компании надо использовать:
1. Контроль доступа (ролевая модель)
2. Маскирование конфеденциальных данных
3. Хранить логи WARNING/ERROR минимум 30 дней, INFO — 7 дней.
4. Хранить Аудит доступа
5. Регулярно проверять логи аудита на подозрительную активность.

3. Проработайте необходимые мероприятия для превращения системы сбора логов в систему анализа логов

Настроить алёрты на панели Grafana для отслеживания критических показателей
Настроить алерты для ERROR-логов .
Настроить алёрты в Prometheus.

Настроить алерты:
 по порогам (например, если CPU больше 90 процентов за 10 минут)
 по изменению частоты возниконовения событий (например, по сравнению частоты возникновения за налогичный период предыдущего бизнес-дня), 
 по возникновению событий (например, ошибки). 














### **Сбор логов в компании "Александрит"**  
Для эффективного мониторинга и поиска проблем необходимо централизованно собирать логи из всех ключевых систем.  

---

## **1. Системы, требующие сбора логов**  
| **Система**               | **Компоненты**                          | **Протокол/Инструмент**          |  
|---------------------------|----------------------------------------|----------------------------------|  
| **Онлайн-магазин**        | Vue (фронтенд), Spring Boot (бэкенд)   | JSON-логи → ELK/Fluentd          |  
| **CRM**                   | Vue (фронтенд), Spring Boot (бэкенд)   | Syslog → OpenTelemetry Collector |  
| **MES**                   | React (фронтенд), C# (бэкенд)          | Windows Event Log → Fluent Bit   |  
| **RabbitMQ**              | Брокер очередей                        | AMQP-логи → Promtail + Loki      |  
| **API для внешних продавцов** | Spring Boot (Java)                  | HTTP-access.log → Grafana Loki   |  
| **Базы данных**           | PostgreSQL/MySQL (Yandex Managed DB)   | Slow Query Log → ELK             |  
| **Серверы инфраструктуры** | Nginx, EC2 (Yandex Cloud)            | Syslog → Vector/Logstash         |  

---

## **2. Список необходимых логов**  
### **А. Логи уровня `INFO` (информационные события)**  
1. **Онлайн-магазин**:  
   - Создание заказа (`INITIATED` → `FILE_UPLOADED` → `SUBMITTED`).  
   - Запросы к API MES для расчёта стоимости.  
   - Ответы от MES (время обработки, статус `PRICE_CALCULATED`).  

2. **CRM**:  
   - Изменение статуса заказа (например, `MANUFACTURING_APPROVED`).  
   - Отправка/получение сообщений через RabbitMQ.  

3. **MES**:  
   - Старт/завершение производства (`MANUFACTURING_STARTED` → `COMPLETED`).  
   - Запросы к базе данных для дашборда.  

4. **RabbitMQ**:  
   - Доставка сообщений (confirmations).  
   - Создание/удаление очередей.  

5. **API**:  
   - Входящие запросы от внешних продавцов.  
   - Ответы MES (успешные/отклонённые).  

---

### **Б. Логи уровня `WARNING` (потенциальные проблемы)**  
1. **Онлайн-магазин**:  
   - Повторные запросы к MES из-за таймаута.  
   - Заказы, зависшие в статусе `SUBMITTED` > 10 минут.  

2. **CRM**:  
   - Сообщения RabbitMQ, обработанные с задержкой > 1 мин.  
   - Несоответствие статусов (например, заказ `APPROVED`, но MES его не получил).  

3. **MES**:  
   - Медленные SQL-запросы (> 500 мс).  
   - Заказы, застрявшие в `PRICE_CALCULATED` > 30 минут.  

4. **RabbitMQ**:  
   - High memory usage (> 80%).  
   - Недоставленные сообщения (NACK).  

5. **API**:  
   - Частые 429 (Too Many Requests) от клиентов.  
   - Невалидные 3D-модели (отклонённые MES).  

---

### **В. Логи уровня `ERROR` (критические сбои)**  
1. **Онлайн-магазин**:  
   - Ошибки оплаты (например, `PaymentFailed`).  
   - 500-е ошибки при отправке заказа.  

2. **CRM**:  
   - Потеря сообщений RabbitMQ (недоставка в MES).  
   - Сбои при изменении статуса заказа.  

3. **MES**:  
   - Ошибки расчёта стоимости (например, `CalculationFailed`).  
   - Дашборд не грузится (timeout базы данных).  

4. **RabbitMQ**:  
   - Queue overflow (очередь переполнена).  
   - Crash брокера.  

5. **API**:  
   - 5xx-ошибки при обработке заказов.  
   - Неавторизованные запросы (401/403).  

---

## **3. Дополнительные логи**  
### **А. Системные метрики (для инфраструктуры)**  
- **Серверы**: CPU, RAM, Disk I/O.  
- **Базы данных**: Deadlocks, Replication Lag.  
- **RabbitMQ**: Queue length, Consumer count.  

### **Б. Аудит безопасности**  
- Попытки входа с неверными паролями.  
- Подозрительные API-запросы (например, SQL-инъекции).  

---

## **4. Инструменты для сбора и анализа**  
- **Логи**: ELK Stack (Elasticsearch + Logstash + Kibana) или Grafana Loki.  
- **Метрики**: Prometheus + Grafana.  
- **Трейсы**: Jaeger/Tempo + OpenTelemetry.  

---

## **Итог**  
| **Тип лога**  | **Примеры**                                      | **Где использовать**                |  
|---------------|-------------------------------------------------|-------------------------------------|  
| `INFO`        | Статусы заказов, успешные API-запросы          | Анализ бизнес-процессов             |  
| `WARNING`     | Задержки в RabbitMQ, медленные SQL-запросы     | Предотвращение сбоев                |  
| `ERROR`       | Падение MES, потеря заказов                    | Срочные инциденты                   |  

**Рекомендации:**  
1. Настроить алерты для `ERROR`-логов (например, в PagerDuty).  
2. Хранить логи `WARNING`/`ERROR` минимум 30 дней, `INFO` — 7 дней.  
3. Регулярно проверять логи аудита на подозрительную активность.  

Этот подход поможет быстро выявлять проблемы, снижая количество жалоб клиентов. 🚀
